# PR レビュー対応

PR のレビューコメントを読み取り、対応方針を判断した上で修正・コミット・プッシュし、各コメントに返答する。

## 入力

$ARGUMENTS

## 手順

1. **PR の確認**

   - PR 番号または URL から内容を取得（引数で渡された場合はそれを使う）
   - 引数がない場合は現在のブランチから PR を特定する
   - PR の概要・変更内容・ベースブランチを把握する

2. **レビューコメントの収集**

   - `gh pr view --comments` や `gh api` でレビューコメントをすべて取得する
   - インラインコメント（コードの特定行への指摘）も含める
   - 対応済みのコメント（resolved / outdated / 自分が返信済みのもの）は除外する
   - コメントごとに以下を整理する:
     - コメント ID
     - 指摘内容の要約
     - 対象ファイル・行（インラインの場合）

3. **対応方針の判断**

   各コメントに対して、以下のいずれかを判断する:

   - **修正する**: 指摘が妥当であり、コードを変更する
   - **修正しない**: 意図的な設計である、スコープ外である、指摘の前提が異なるなど、変更しない理由がある

   判断が難しいものや、仕様・要件に関わるものは実行者に確認する。

   **対応方針が決まったら、作業を始める前に実行者に方針を共有する。**

4. **修正の実施**

   「修正する」と判断したコメントへの対応をコードに反映する。

   **コミットの分け方:**

   - コメント 1 件 = コミット 1 件である必要はない
   - 意味のある変更のまとまりを 1 コミットにする
   - 例:
     - 同じ関数に対する複数の指摘 → 1 コミット
     - 別々の責務に関わる修正 → 別々のコミット
     - リネームと振る舞いの変更が混在 → 別々のコミット
   - コミットメッセージは変更内容を簡潔に記述する

5. **リモートへのプッシュ**

   - 修正コミットをリモートブランチにプッシュする
   - プッシュ後、コミットハッシュを確認・記録する

6. **レビューコメントへの返答**

   各レビューコメントに対して返答する。

   **修正した場合:**

   - 何をどう修正したかを簡潔に説明する
   - 対応したコミットのハッシュを含める
     - GitHub がリンクに変換するため、バッククォートで囲まない
     - ハッシュの前後に文字を直接繋げない（空白または行頭・行末でないとリンクにならないため）
     - 例: `対応コミット: abc1234` のように、ハッシュの前後は空白か改行で区切る
   - 複数コメントをまとめて 1 コミットで対応した場合は、そのコミットハッシュを各コメントに記載する

   **修正しない場合:**

   - 修正しない理由を明確に説明する
   - 設計上の意図・前提・スコープなどを丁寧に伝える

   **返答の形式例（修正した場合）:**

   > ご指摘ありがとうございます。〇〇を△△に変更しました。
   >
   > 対応コミット: abc1234

   **返答の形式例（修正しない場合）:**

   > ご指摘ありがとうございます。こちらは〇〇という理由から意図的にこの実装にしています。
   > △△という背景があるため、今回は変更しない方針としました。

7. **完了の報告**

   - 対応したコメント数・修正コミット数を実行者に報告する
   - 修正しなかったコメントがある場合は、その理由をまとめて報告する
