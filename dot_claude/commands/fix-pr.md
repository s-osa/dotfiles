# PR チェック修正

PR で失敗している CI やその他のチェックを調査・修正し、マージ可能な状態にする。

## 入力

$ARGUMENTS

## 手順

1. **PR の確認**

   - PR 番号または URL から内容を取得（引数で渡された場合はそれを使う）
   - 引数がない場合は現在のブランチから PR を特定する
   - PR の概要・変更内容・ベースブランチを把握する

2. **失敗しているチェックの確認**

   - `gh pr checks` で各チェックのステータスを確認する
   - 失敗しているチェックのログを取得して原因を把握する
   - 複数のチェックが失敗している場合は、それぞれ整理する

3. **原因の調査**

   - エラーメッセージ・ログをもとにコードベースを調査する
   - 以下のような原因を想定して調べる:
     - テストの失敗（実装のバグ、テストの前提条件の問題）
     - 型エラー・コンパイルエラー
     - lint / フォーマットの違反
     - ビルドの失敗
     - 環境・依存関係の問題
   - 原因が特定できない場合や、修正方針の判断が難しい場合は実行者に確認する

4. **修正の実施**

   特定した原因に対して修正を行う。

   **コミットの分け方:**

   - 意味のある変更のまとまりを 1 コミットにする
   - 例:
     - 同じ原因による複数ファイルの修正 → 1 コミット
     - テスト修正とプロダクションコードの修正が別の原因 → 別々のコミット
     - lint 修正とロジック修正 → 別々のコミット
   - コミットメッセージは変更内容を簡潔に記述する

5. **ローカルでの確認**

   - 修正後、該当するチェックをローカルで実行して問題が解消されていることを確認する
   - CLAUDE.md やプロジェクトの設定からコマンドを確認する

6. **リモートへのプッシュ**

   - 修正コミットをリモートブランチにプッシュする
   - CI が再実行されるのを確認する

7. **結果の確認と報告**

   - チェックが通過したことを確認する（`gh pr checks` で確認）
   - 失敗が残っている場合は手順 3 に戻る
   - すべて通過したら実行者に報告する
