# Issue 作成

ユーザーから受け取った内容を整理し、**課題を明らかにして** GitHub Issue を作成する。

このコマンドの目的は「何が問題で、なぜ解決が必要か」を明確にすることであり、解決方法の詳細な調査は行わない（それは `/solve-issue` の役割）。

## 入力

$ARGUMENTS

## 手順

1. **入力の解析**

   - ユーザーの依頼内容・バグレポート等を読み取る
   - URL が含まれている場合:
     - WebFetch を試みて内容を参考にする
     - **読み取れない場合（認証が必要な Slack, Google Docs 等）は無理に読まず、参考 URL として Issue にリンクを残す**

2. **課題の明確化**

   以下を明らかにすることに集中する:

   - **何が問題か**: 現状どうなっていて、何が困っているのか
   - **なぜ解決が必要か**: 背景・モチベーション
   - **完了条件**: 何をもって Done とするか

   不明確な点があればユーザーに質問して確認する。

3. **Issue の作成**

   - タイトル: 簡潔に何をするかを記述（例: 「ログイン画面にパスワードリセット機能を追加」）
   - 本文の構成:
     - **背景**: なぜこの Issue が必要か
     - **課題**: 現状の問題点
     - **やりたいこと**: 具体的な要件
     - **完了条件**: 何ができたら完了か
     - **解決の方向性**（あれば）: 「こういうアプローチが考えられる」程度の軽い言及。コード調査や詳細設計は書かない
     - **フェーズ計画**（複数ステップがある場合）:
       - 3つ以上のステップ・フェーズがある場合は必ず記載
       - 各フェーズの概要をリストアップ（例: Phase 1: 調査、Phase 2: 実装、Phase 3: テスト）
       - 「各フェーズはサブイシューとして切り出して進める」旨を明記
     - **参考資料**（あれば）: URL（読み取れなかったものも含む）、スクリーンショットなど
   - ラベル: リポジトリに既存のラベルから適切なものを選んで付与する（なければ省略）。**新しいラベルを作成しないこと**

4. **結果の報告**
   - 作成した Issue の URL を報告する
   - フェーズ計画がある場合は、サブイシュー化の予定を伝える
   - `/solve-issue #番号` で作業を開始できる旨を案内する
